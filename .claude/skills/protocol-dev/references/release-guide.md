# 版本发布流程规范

## 触发条件

当用户说"准备发布新版本"、"我要发布新版本"、"准备发行版本"等时，自动执行版本发布流程。

## 任务一：查找最新版本号

### 步骤 1：获取当前分支信息

```bash
git branch --show-current
```

### 步骤 2：查找版本信息

1. **从 git tag 获取最新版本**：
   ```bash
   git tag --sort=-v:refname | head -10
   ```

2. **从 Cargo.toml 获取当前版本**：
   ```bash
   grep '^version' Cargo.toml
   ```

3. **比较确定最新版本号**

### 步骤 3：输出结果

输出格式：

```
最新版本号：2026.2.6
最新 git tag：v2026.2.6
Cargo.toml 版本：2026.2.6
```

## 任务二：生成更新日志

### 步骤 1：确定版本范围

1. **找到上一个版本发布的 tag**：
   ```bash
   git tag --sort=-v:refname | head -2
   ```

2. **获取从该 tag 到当前 HEAD 的所有提交**：
   ```bash
   git log <上一个版本tag>..HEAD --oneline
   ```

### 步骤 2：分析提交信息

1. **按类型分类提交**：
   - `feat`：新功能
   - `fix`：问题修复
   - `perf`：性能优化
   - `refactor`：重构
   - `chore`：其他（排除 release 类型）

2. **识别同一功能的多个提交**：
   - 通过 `scope` 识别（如 `feat(admin)`、`feat(proxy)`）
   - 通过描述关键词识别（如"实现"+"优化"、"新增"+"改进"）

3. **整合相邻的同一功能提交**：
   - 如果相邻提交的 scope 相同且描述相关，合并为一条
   - 合并时只保留最终效果描述

### 步骤 3：生成更新日志

#### 格式结构

```markdown
chore(release): 🔖 发布 v[版本号] 版本 - [版本主题描述]

- [功能点1]：[简洁描述功能和效果]
- [功能点2]：[简洁描述功能和效果]
- [功能点3]：[简洁描述功能和效果]
```

#### Header 格式

- `chore(release): 🔖 发布 v[版本号] 版本 - [主题描述]`
- 主题描述：一句话概括本次更新的核心亮点
- 示例：`chore(release): 🔖 发布 v2026.2.7 版本 - Credential 管理增强与性能优化`

#### Body 格式

- 使用无序列表（`- `）
- 每个条目一行，简洁描述功能和效果
- 格式：`[功能名称]：[效果描述]`

#### 内容整合规则

1. **合并规则**：
   - 相邻提交如果 scope 相同且描述相关，合并为一条
   - 同一功能的多个提交（如"实现"+"优化"+"修复"）合并为一条
   - 只写最终效果，不写开发过程

2. **过滤规则**：
   - ❌ 不包含开发过程中的问题（如"修复新功能的 bug"）
   - ❌ 不包含探索性提交（如"尝试"、"测试"、"调试"）
   - ❌ 不包含内部技术细节（如"重构代码结构"）
   - ✅ 只写最终交付的功能和效果
   - ✅ 突出用户价值和体验改进

3. **描述规范**：
   - 使用简体中文
   - 简洁明了，突出用户价值
   - 避免过于底层的技术术语

### 步骤 4：生成版本主题

根据整合后的功能点，生成一个简洁的主题描述：

- 如果主要是新功能：`[核心功能]与[其他改进]`
- 如果主要是修复：`[核心修复]与稳定性优化`
- 如果功能较多：`[主要功能1]、[主要功能2]与[其他改进]`

示例：
- `Credential 批量导入与 Token 管理优化`
- `SSE 流式响应修复与性能提升`
- `Admin UI 增强与多 Credential 调度优化`

### 步骤 5：版本号更新

**需要更新的文件**：
- `Cargo.toml` 中的 `version` 字段

**版本号格式**：遵循项目现有的版本号格式（如 `2026.2.6`）

## 完整工作流程示例

1. **用户说**："准备发布新版本"

2. **执行任务一**：
   - 查找当前分支
   - 查找版本信息：git tag + Cargo.toml
   - 输出：最新版本号

3. **执行任务二**：
   - 找到上一个版本 tag
   - 获取提交列表
   - 分析并整合提交
   - 生成更新日志草稿

4. **展示结果**：
   - 显示最新版本号
   - 显示生成的 Git commit 更新日志草稿
   - **提供版本号更新建议**
     - 建议新版本号
     - 说明需要更新 Cargo.toml
   - 等待用户确认或修改
   - **结尾提示**（一次性告知所有后续操作）：
     ```
     请确认内容是否需要调整。确认后：
     1. 更新 Cargo.toml 中的版本号为 {版本号}
     2. 手动提交 release commit
     3. 提交后打版本标签：`git tag v{版本号}`
     ```

5. **用户确认后**：
   - 用户授权打标签后执行：`git tag v{版本号} {commit hash}`
   - 验证：`git tag -l v{版本号}`

# 分支合并规范

## 触发条件

用户提出分支合并相关需求，包括但不限于：
- "把 main 合并到当前分支"
- "合并 xxx 分支"
- "同步 main 的最新代码"
- "merge main"

## 核心原则

1. **禁止直接 commit**：合并完成后只生成 commit 信息，由用户手动提交
2. **冲突必须停下**：遇到冲突时必须先分析原因、给出方案、等待用户授权后才能解冲突
3. **先谋后动**：合并前必须先分析两个分支的分歧情况

## 工作流程

### 第一步：合并前分析

执行以下命令了解分支分歧情况：

```bash
# 查看两个分支的分歧 commit
git log --oneline <当前分支>...<目标分支> --left-right

# 查看当前分支状态
git branch -v
```

**输出要求**：
- 简要说明目标分支有多少个新 commit 需要合并进来
- 简要说明当前分支有多少个独有 commit
- 概括两边的主要改动方向

### 第二步：执行合并（强制 --no-commit）

```bash
git merge <目标分支> --no-commit --no-ff
```

**强制要求**：
- 必须使用 `--no-commit`，禁止自动提交
- 必须使用 `--no-ff`，保留合并历史

### 第三步：处理合并结果

#### 情况 A：无冲突

1. 告知用户合并成功，无冲突
2. 执行 `git diff --cached --stat` 查看合并引入的变更
3. 按照 commit-guide 中的 merge commit 规范生成提交信息

#### 情况 B：有冲突

**🚨 必须停下来，禁止直接解冲突**

1. **定位冲突文件**：
   ```bash
   # 查找冲突标记
   grep -rn "^<<<<<<<\|^=======\|^>>>>>>>" <冲突文件>
   ```

2. **分析每处冲突**：
   - 读取冲突区域的上下文
   - 说明 HEAD（当前分支）的内容是什么
   - 说明目标分支的内容是什么
   - 分析冲突原因（同位置修改、相邻插入、逻辑冲突等）

3. **给出合并方案**：
   - 对每处冲突提出解决策略（保留双方、取其一、手动融合）
   - 说明理由

4. **等待用户授权**：明确告知"确认方案没问题的话我就执行解冲突"

5. **执行解冲突**（用户授权后）：
   - 按方案修改冲突文件，删除所有冲突标记
   - 验证无残留冲突标记：`grep -c "^<<<<<<<\|^=======\|^>>>>>>>" <文件>`
   - 标记文件为已解决：`git add <冲突文件>`
   - 执行 `git status --short` 确认合并状态正常

6. **生成 commit 信息**：按照 commit-guide 中的 merge commit 规范生成

### 第四步：生成 Commit 信息

合并完成后，按照 commit-guide 的 merge commit 专用规范生成提交信息。

**关键差异**：merge 状态下必须使用 `git diff --cached` 系列命令（而非 `git diff HEAD`），因为 merge 状态下 HEAD 仍指向合并前的 commit。

## Cargo.lock 冲突特殊处理

Rust 项目中 `Cargo.lock` 是最常见的冲突来源，处理时注意：

- **通常选择目标分支版本**后重新 `cargo update` 即可
- 如果双方都更新了不同的依赖，可以先接受一方，再手动 `cargo update -p <package>` 更新另一方的依赖
- 解冲突后执行 `cargo check` 确认依赖解析正确

## 禁止事项

- ❌ 不使用 `--no-commit` 直接合并
- ❌ 遇到冲突不分析直接解决
- ❌ 合并后直接执行 `git commit`
- ❌ 使用 `git merge --abort` 放弃合并（除非用户明确要求）
- ❌ 使用 `git checkout --theirs` 或 `git checkout --ours` 粗暴解决冲突（除非分析后确认合适）
- ❌ 跳过冲突标记残留检查

# Bug 调试规范 (The "DEBUG-FIRST" Rule)

当用户报告程序中的 Bug 时，**严禁凭直觉猜测问题原因或直接修改业务逻辑**。必须严格遵循以下调试流程。

## 第一步：分析问题现象

1. **明确异常行为**：用户描述的实际发生了什么
2. **明确预期行为**：按照设计应该发生什么
3. **定位问题范围**：确定涉及的代码模块、执行路径和可能的触发条件

## 第二步：提出调试方案（必须等待用户批准）

**🚨 禁止直接向源代码写入调试信息！** 必须先在对话中向用户描述完整的调试方案。

### 调试方案必须包含以下内容

1. **问题分析**：简述对问题的理解和初步判断
2. **调试思路**：解释为什么需要在这些位置添加日志，以及如何通过日志定位问题
3. **目标文件清单**：列出需要添加调试信息的文件及具体位置
4. **日志设计**：
   - 使用统一前缀（如 `🔬 [模块名]`）便于过滤
   - 明确要记录的变量、状态、条件判断结果
   - 标记执行顺序和时机
5. **明确请求批准**：最后一句必须是"确认后我开始添加调试代码"

### 方案设计原则

- 每个关键分支都要有日志覆盖
- 日志要能明确区分"进入了哪个分支"
- 异步操作的开始和结束都要记录
- 状态变化的前后值都要记录

## 第三步：插入调试代码（仅在用户批准后执行）

收到用户明确批准（如"可以"、"执行"、"开始吧"）后，向代码中添加调试日志。

### 日志代码规范

```rust
// ✅ 信息完整的日志
tracing::debug!("🔬 [TokenManager.refresh] 开始刷新 - credential_id={}, token_age={:?}", id, age);

// ✅ 条件分支日志
if token.is_expired() {
    tracing::debug!("🔬 [TokenManager] token 已过期 - expired_at={:?}", token.expires_at);
} else {
    tracing::debug!("🔬 [TokenManager] token 仍有效 - remaining={:?}", token.remaining());
}

// ✅ 异步操作日志
tracing::debug!("🔬 [Provider] 发送请求开始");
let result = client.post(url).send().await;
tracing::debug!("🔬 [Provider] 请求完成 - status={:?}", result.as_ref().map(|r| r.status()));
```

### 日志插入位置

- 函数入口和出口
- 条件判断的每个分支
- 循环体内的关键操作
- 异步操作的启动和回调（`.await` 前后）
- 状态变量的修改前后

## 第四步：用户验证与日志分析

1. **请用户复现问题**：运行程序，触发 Bug
2. **收集控制台日志**：用户复制完整的日志输出
3. **分析执行路径**：
   - 对比实际执行路径与预期路径
   - 定位第一个异常点（变量值不符合预期、分支未按预期执行）
   - 分析异常原因（逻辑错误、状态不一致、时序问题等）
4. **明确问题根因**：基于日志证据得出结论，而非猜测

## 第五步：修复与清理

1. **提出修复方案**：
   - 说明问题根因
   - 描述修复思路
   - 列出需要修改的代码
   - **等待用户确认**

2. **执行修复**（用户批准后）：
   - 修改业务逻辑
   - **保留调试日志**（不要在此步骤清理）

3. **用户验证**：
   - 请用户运行程序验证问题是否已修复
   - 等待用户反馈结果

4. **清理调试代码**（🚨 必须等待用户明确指令）：
   - 用户确认问题已修复后，**不要主动询问是否清理**，等待用户明确指令
   - 只有用户明确说"清理调试日志"、"删除日志"等指令后，才能删除调试代码
   - **严禁在功能开发完成后自动清理调试代码**
   - **严禁在执行修复时顺便清理调试代码**
   - 调试日志的保留或清理完全由用户决定时机

## 🚫 严禁行为

- ❌ 未经用户批准直接添加调试日志
- ❌ 不看日志就凭猜测修改逻辑
- ❌ 连续尝试多个"可能的修复"而不通过日志验证（禁止"改一下试试"的盲目尝试）
- ❌ 未经用户明确指令直接清理调试代码（即使问题已修复）
- ❌ 在执行修复时顺便删除调试日志
- ❌ 跳过方案描述阶段直接修改代码
- ❌ 主动询问用户是否清理调试日志（应该等用户主动提出）
- ❌ 功能开发完成后自动清理调试日志

## ✅ 正确示例流程

**用户**："代理请求偶尔返回 502"

**AI 回复**：

```
问题分析：代理转发请求到 Kiro API 时偶尔返回 502，可能是上游连接问题或 token 过期。

调试思路：在请求转发链路的关键节点添加日志，追踪：
1. 请求到达时的 credential 选择
2. Token 状态检查结果
3. 上游请求的发送和响应
4. 重试逻辑的触发条件

需要添加调试信息的文件：
* `src/kiro/provider.rs`：在请求发送前后记录状态
* `src/kiro/token_manager.rs`：在 token 获取时记录有效性

日志设计：
* 使用前缀 `🔬 [Provider]` 和 `🔬 [TokenManager]` 便于过滤
* 记录关键变量：credential_id、token 过期时间、HTTP 状态码
* 标记执行顺序：选择 credential → 获取 token → 发送请求 → 处理响应

确认后我开始添加调试代码。
```

**用户**："可以"

**AI**：[添加调试代码]

**用户**：[粘贴日志] "看到 token 获取成功了，但上游返回 502"

**AI**：

```
根据日志分析，问题定位：
Token 有效，但上游 Kiro API 返回 502。

问题根因：在高并发场景下，同一 credential 的多个请求共用同一 token，
但 Kiro API 对单 token 有并发限制。

修复方案：
在 provider 中添加 per-credential 的请求队列，限制单 credential 并发数。

确认后我执行修复。
```

**用户**："确认"

**AI**：[执行修复，保留调试日志] "修复已完成，请运行程序验证问题是否解决。"

**用户**："问题解决了"

**AI**："好的，修复已完成。"

[用户验证一段时间后]

**用户**："清理调试日志"

**AI**：[删除所有调试日志] "调试日志已清理。"

## 核心原则总结

1. **先分析，后行动**：永远不要跳过分析直接修改代码
2. **日志驱动调试**：所有结论必须基于日志证据
3. **用户批准制**：任何代码修改前都必须获得用户明确批准
4. **清理需指令**：调试日志的清理必须等待用户明确指令（如"清理调试日志"），不要主动询问，不要自动清理
